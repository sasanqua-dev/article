## TL;DR - この記事の要約

開発に携わっていたシステムの権限制御を部分的に自前で持ってるRBACからOIDC経由で降ってくる情報を元にするRBACに変えてみました。
自前で持っていたスコープベースのRBACよりも圧倒的に少ないコード量で記述することができた一方、拡張性がそこまで大きいわけではないためアプリケーションの規模に応じた方法を検討することが重要だと思いました。
（薄い感想ではありますが、正直実際に運用してみないと適切なものを選ぶのは難しそうだなと思います）

## RBAC / OIDC

ここでのRBACはRole Based Access Controllをさしています。具体的にはロールというオブジェクトに対して権限を付与し、そのオブジェクトに対してユーザーを紐づける方法でユーザーの権限を管理する手法です。
また、OIDCはOpen ID Connectという認証のための規格、ですかね。（間違っているかもです）
認証フローや実際のコードとしてはOAuth2と同じような流れなので多くのフレームワークの認証ライブラリが対応しています。
OAuth2は認可のための仕組みです（最近は認証のための手段として用語が使われていることもありますが、）、一方でOIDCはOauth2を拡張した認証のための仕組みです。

認証周りの文脈として「SSO - Single Sign On」という用語がよく出てきますが多くのアプリでのOIDCの導入動機は既存のサービスでのログイン情報を用いて自身のアプリケーションへのログインを行うことだと思います。SSOはまさにこれらの「それぞれのサービスでのログインを行うのではなく１つの場所へのログインで複数サービスを使える状態」だと自分は認識しています。

## 既存の構成

今回の話題に上がるアプリではこれまでAuth0のサービスを用いてGoogleログインが使われていました。
組織（ここでの組織はアカウント発行単位）内のアカウントは管理権限を持つユーザーと持たないユーザーがいる状態のためロールを用いてアクセス制御を行っていました。
このアプリで使われていたより細かい権限の制御方式は「スコープ」と「操作」を定義し

```md
hpc:{scope}.{operation}
// hpcは本アプリ上での権限制御基盤の名称です
```

の形式で定義された権限制御をエンドポイントごとに定義し、それに合致する権限を持っているロールを持っているユーザーからのリクエストであれば通す方式でした。
AWSのIAMなどと同じような仕組み（というかそれを模倣して設計した）です。

## 問題点

この運用自体に不都合は特にありませんでした。細く権限の制御もできるし、コード側もリクエストする権限がわかりやすくなるし。
しかしながら、実運用をして見ると別に細かい権限制御を求めていたわけではないことに気づきました。
権限設定がめんどくさくなってユーザーは管理者権限のみを振るのです。
一方で、この権限の維持には大量のソースコードとテーブルが絡んでいます。
特に使わない機能を残しておくことは技術的負債につながるので一気にリファクタすることにしました。

## OIDCをベースにした権限管理の実装

この環境では、最近「Pocket Base」というセルフホスト可能なOIDC Providerのホストを始めていたため、Provider上のグループ機能を用いて権限制御を行うことにしました。

また、権限の区分についてもscope x actionのマトリクスではなく必要な権限のみを定義して、そこからエンドポイント側が要求したい権限を選び、権限制御を行う方式をとっています。

### 要求権限とユーザーの種別を型で定義

```python
class HPCUserTypeScheme(Enum):
    # OIDC認証
    Architect = "Architect"
    Manager = "Manager"
    Member = "Member"
    # 通常ユーザー
    User = "User"
```

管理権限を持ち得るのは組織内の人間（= OIDC経由でログインしている）という前提をもとにユーザーを４パターンで表現しています。
権限管理側の型定義はセキュリテイ上の都合で掲載できませんが、同じように定義しユーザー種別と要求権限の種別で表現することで全ての権限制御のルールの定義をユースケース層で行なっています。
コード量も少なく、視覚的にもわかりやすいためコーディングに精通していない人でも簡単に変更のPR作成が行えるようになりました。

## 最後に

権限管理とアカウント管理は共に、組織の大きさやデジタル化の進捗によって必要なものが大きく変わってくるものだと思います。
過度に複雑なものを導入せずに、それぞれのフェーズに合った実装をしていきたいと思いました。
